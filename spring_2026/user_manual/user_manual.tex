\section{Quick Start Guide}

This drone is designed to be used in-lab with just the Wingflasher application and the required hardware. This short initial guide is intended to move the user from having a drone constructed to being ready to fly. More details on how to accomplish any of the steps listed below can be found later in the document.

\begin{enumerate}
\item Assembling the hardware can be challenging, and thus does not belong in this guide. For this reason, the Quick Start Guide assumes that you have followed the Drone Construction section located below. 

\item Once this is accomplished, download the app from the \href{https://github.com/Team-Wingfeather/fcl_senior_design/releases}{releases} section of the \href{https://github.com/Team-Wingfeather/fcl_senior_design}{project GitHub page}.

\item Using a USB-C cable that supports data transfer, plug the drone into an available port on the host machine.

\item Launch the app, navigate to the \texttt{Programming} tab, and press ``Flash Drone''

\item Disconnect the drone from the computer.
\end{enumerate}

The drone is now ready for flight testing. It has been flashed with a basic flight script and parameters that can now be found next to the flashing application under \texttt{config}. The testing procedure is discussed in more detail in a later section, but the basic steps are:

\begin{enumerate}
\item If using the test apparatus, fasten the drone to the apparatus.
\item Place the drone upright and flat on the table (while still on the apparatus if applicable) and plug in the 7.4V LiPo battery.
\item Connect to the drone's Wifi. (The password is "Password").
\item Run \texttt{test.py}. Press \texttt{ctrl+C} in the terminal to stop the drone.
\end{enumerate}


\section{Drone Construction}

\subsection{Ordering Components}
\subsubsection{PCB}

Essential files for working with the Wingfeather circuit board are located in the \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/main/pcb}{\texttt{pcb} directory}. This includes both the editable EasyEDA files and Gerber files for ordering.

Ordering printed circuit boards is made simple by \href{https://jlcpcb.com/}{JLCPCB}, which is also directly accessible from an EasyEDA project. In general, the cheapest settings can be used for ordering.

\subsubsection{Motors}
The current motors used by the Wingfeather project are 720 coreless brushed DC motors. Most of these found on Amazon are about the right size, but stock tends to fluctuate. The current ones being used are found \href{https://www.amazon.com/dp/B0BBR16BK3}{here}.

\subsubsection{Propellers}
The current drone propellers can be found \href{https://www.amazon.com/uxcell-Propellers-Walkera-Ladybird-Quadcopter/dp/B07Y6LP7QP/ref=sr_1_2?sr=8-2}{here}. Many different types of propellers can be used; the important part is that the shaft hole diameter is around 1mm and the length is about 55mm.

\subsection{Time of Flight Sensors}

The current time of flight sensors in use on the Wingfeather drone are the VL53L1. These can be purchased with many different breakout board options, but as long as the pin order and dimensions match the main printed circuit board, they should work. This is a 6-pin configuration in the order \texttt{VCC}, \texttt{GND}, \texttt{SCL}, \texttt{SDA}, \texttt{GPIO1}, and \texttt{XSHUT}. One version we used can be found \href{https://www.amazon.com/DWEII-Measurement-Extension-Compatible-Raspberry/dp/B09V4DS888?tag=bngsmtphsnus-20&linkCode=df0&hvadid=79989669526635&hvnetw=s&hvqmt=e&hvbmt=be&hvdev=c&hvlocint=&hvlocphy=93603&hvtargid=pla-4583589133736495&psc=1}{here}, but there are also more cost effective variations available such as \href{https://www.ebay.com/itm/335880426834?_skw=vl53l1x&itmmeta=01KBQMDZ36P3727PVEQP04QS72&hash=item4e34089552%3Ag%3AXFkAAOSwfHtn42--&itmprp=enc%3AAQAKAAAA4FkggFvd1GGDu0w3yXCmi1fhVCjvI%2B6gXNmTwNwV%2FI08OUnLs7IEiPrwcSOQ1Qxs53Oeh6PQi9Sw%2FcOp7kobl6Ifs5FaNnmMREjD2HPECAXtjsOyEYjA2zgul1N4Th7ZBKM5KZQoCGErFpuL91vwmSvskTtpQBmZ2eRHIJpe1Q2J%2BLSHlZlbRMI7wtpZmEi1SievedcCKPnAg5mRAvRSu5SnSRFxJCjBVXbOwuE5daF3NWeITinfEQQS144ojU420kFEqwyifo2z6LeLFOEN2KV21gZf9PkgrligSrAzT1ks|tkp%3ABFBM3vG39N1m&var=545425037697}{this one}. \subsection{Printing the Frame}
To print the current frame, you need to install Bambu Labs software (obtained from Cedarville software). This provides all the necessary tools to print the frame. Note that the frame should be printed upside-down out of PLA or PETG.

\subsection{Assembling the components}

Once the necessary components have been obtained, assembling the drone is fairly straightforward.
\begin{enumerate}
\item Solder the battery connector of choice to the bottom of the board. The plastic part of the connector can slide off of the metal part, so it can be helpful to super-glue the plastic to the PCB before soldering. Note that there are two connector options: JST-PH and JST-XH. To date, the team has been using JST-PH connectors for their minimal profile. Be careful to match the polarity of the batteries to the polarity of the board, marked with $+/-$ symbols near the connector.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4404}
    \caption{Drone Charge Port}
    \label{fig:ChargePort}
\end{figure}
\item Using header pins, solder 2 time-of-flight sensor breakout boards onto the main board---one on top and the other on the bottom. These should use the 6-pin header slots, and be soldered onto the face with the printed labels.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4405}
    \caption{Mounted Time-of-Flight Breakout Board}
    \label{fig:tofBreakout}
\end{figure}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4406}
    \caption{PCB Screwed to Frame}
    \label{fig:PCBMount}
\end{figure}
\item Place the PCB on the frame and use 7 M2 screws to connect them. Note that the orientation of the PCB is determined by which corner does not have a hole for a screw. The ``front'' of the drone is the side with the USB connection.

\item Connect the propellers onto the motors. Make sure to select where each motor will go and match the propeller direction with the direction indicated around the screw nearest to that motor. Pushing gently onto the table can help affix the propellers to the shafts. Finally, it  is conventional to use a different color of propellers to designate the front of the drone, depending on the model purchased.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4407}
    \caption{Match Propeller Direction to Indicator}
    \label{fig:PropDir}
\end{figure}
\item Slide the 4 motors into the designated recesses. It may prove easier to solder the motor connections before gluing the motors in.
\item Solder each motor to the nearest 2 through holes on the circuit board. Note that polarities of the annular rings alternate around the drone. This is so that the motors across from each other spin the same direction and adjacent motors spin opposite directions. In practical terms, the square annular ring should always be connected to the same motor terminal ``polarity'' (designated by the color of the wire). Which color may need to be determined by trial and error. For the current motors, the white motor lead should be soldered to the circular annular ring. In the end, make sure the direction each motor spins matches the arrow located around the nearest PCB screw. (This can be verified by flashing the drone using the app and connecting the battery---each motor should spin briefly). Note that it may be helpful to twist the wires from each motor before soldering. After soldering, this twisted pair may be hot glued to the frame to save space. Use caution when using the glue gun and soldering iron, as they can easily melt the 3D-printed frame.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4410}
    \caption{Motor Wires Glued Down}
    \label{fig:MotorWires}
\end{figure}
\item Slide each motor partially out and apply a moderate amount of hot glue under it. Slide the motor back down, and be ready to wipe away excess glue. Affixing the motors well is important so they do not fly off in flight. 
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4408}
    \caption{Hot Glue Applied to Motors}
    \label{fig:HotGlue}
\end{figure}
\item In the future, more time-of-flight sensors may be added to the 5-pin header slots. Additionally, a Velcro strap can be used to affix the battery to the bottom of the board through the designated slot.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4411}
    \caption{Battery Attached to Bottom of Drone}
    \label{fig:BatteryMount}
\end{figure}
\end{enumerate}

\subsection{Using the App}
As mentioned in the Quick Start section, the app can be downloaded from the project \href{https://github.com/Team-Wingfeather/fcl_senior_design}{GitHub page} under \href{https://github.com/Team-Wingfeather/fcl_senior_design/releases}{releases}. Details on the features of each tab are listed below.
\subsubsection{Graphing Tab}
The graphing tab includes the ability to load a CSV file from your system which you will be able to cycle through, as well as various options from Matplotlib to resize, save a photo of, or move graphs around. CSVs are obtained from the drone itself, which has protocols to create them from its own flight data.

\subsubsection{Planning Tab}
The planning tab includes multiple stubs for lab options to be implemented in the future. Current progress on the project has been solely towards simple PID altitude control, but this tab will be more useful as the lab grows more complex.

\subsubsection{Programming Tab}
The programming tab is for setting the PID values to flash to the drone. Currently, the parameter-setting function of this tab is unimplemented. In the future, it should be able to set the altitude position adn velocity P, I, and D values within a reasonable range that is determined experimentally. Additionally, the advanced mode (discussed in the next section) could be augmented to allow for full parameter access.

The ``Flash Drone'' button is functional, and will auto-detect the drone and flash scripts and parameters. These parameters are currently located in a \texttt{config} directory that is created the first time the drone is flashed. Once this is completed, the developer may enter this directory to modify the parameters for flashing. The \texttt{commands.txt} file is currently unused, but an in-progress implementation of the scripting exists on the branch \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/commander-control}{\texttt{commander-control}}.

Note that the drone hardware has been designed so that the user can plug the drone into the computer for flashing without disconnecting the battery. 

There are a few known minor bugs with the flashing system, but disconnecting and reconnecting the drone should solve most of these. In general, do not press start while the drone is connected to the computer.

Flashing the drone with modified firmware via the app can only be accomplished via advanced mode (see below). Each year the lab is performed should only need one version of the firmware, so this simplifies the setup. Changes for automation of the versioning system on this could be made in the future.

\subsubsection{Advanced Mode}
The app has an advanced mode that can be accessed via the file tab on the menu bar. The purpose of this mode is to allow users more features of the drone for debugging, development, and advanced use beyond lab experiments. Currently, this mode only adds an option to the programming tab to allow the user to force a full flash of the drone without regard to the current partition table. 

\section{Drone Development}

\subsection{Development Environment}
The current framework for developing firmware for the Wingfeather is \href{https://docs.espressif.com/projects/esp-idf/en/v5.5.4/esp32/get-started/index.html}{ESP-IDF v5.5.4}. Espressif has recently released IDF v6 and the project could be upgraded to v6 if desired. An installation guide can be found by following the above link.

\subsection{IDE Setup}
For ease of use, you can develop in VSCode with the \href{https://github.com/espressif/vscode-esp-idf-extension/blob/master/README.md}{ESP-IDF Extension}. 
The extension adds buttons to the VSCode toolbar that allow for easy compilation and flashing of the firmware.

If you have a preferred code editor, however, the command line experience is also simple.

\subsection{Basic Repository Structure}

\subsubsection{The drone-firmware directory}
The entire esp-idf project for the drone firmware can be found in the \texttt{drone-firmware} directory. The directory contains the following files:

\begin{itemize}
  \item A \texttt{CMakeLists.txt} file that specifies the project's name and build options.
  \item A partition table that specifies the flash layout of the project.
  \item A main directory
  \begin{itemize}
    \item A CmakeLists.txt file that specifies the entry point, component directory, and required headers for main.
    \item A main.c file that contains the entry point
    \item A linker\_fragment.lf file that specifies some linker rules. This is so that some crazyflie functions get linked correctly in esp-idf.
    \item An idf\_component.yml file that specifies the dependencies of the project.
  \end{itemize}
  \item A components directory that contains all of the components used by the project
  \item The rest of the files at the firmware root are autogenerated by the build system.
\end{itemize}
  
\subsubsection{Other Important Directories}
The flashing application is located in \texttt{wingfeather\_app}, along with its dependencies and instructions. \texttt{frame} and \texttt{pcb} contain the components for these parts of the project, along with instructions on how to use them. \texttt{wireless\_testing\_scripts} contains scripts for running the drone on the testing apparatus (see Basic Drone Testing below). The remainder of the directories contain documentation and resources. This includes records of past work and information for future development. 

\subsection{Basic Drone Testing}
To test the controller:
\begin{enumerate}
\item If using the test apparatus, fasten the drone to the apparatus.
\item Place the drone upright and flat on the table (while still on the apparatus if applicable) and plug in the battery. If the drone is not perfectly level at initialization or during resets, the IMU will not start up properly. It is recommended to already have the plug near the appropriate place before laying it down. This is a safety feature from the firmware that is easier to work with without the apparatus.
\item Flash the PID parameters as instructed in ``Using the App'' and press start. NOTE: Keep the drone flat in case resetting at the end of the flash messes up the IMU.
\item Connect to the drone's Wifi (the password is ``Password'').
\item Run test.py. Press ctrl+c in the terminal to stop the drone. If something goes wrong and ctrl+c does not stop the drone, press the reset button on the drone.
\item To plot logged data, run plotcsv.py or use the app.
\end{enumerate}
Setting up test.py:
\begin{itemize}
\item LOG\_TYPE: Selects which values you want to log.
\begin{itemize}
\item Additional sets of logged values can be made following the formatting of other sets.
\item In log\_callback(), print() and log.write() must be specified. flight\_test must also be updated to add a new set.
\item Values available to log can be found in LogableValues.txt.
\item There is a limit to how much you can log (usually no more than 6 floats)
\end{itemize}
\item JUST\_LOG: Set to 1 if you just want to gather values without running the motors.
\item FLIGHT\_TYPE: Set to 1 for normal flight. Setting to 0 will specify a target thrust, yaw, pitch, and roll (useful for PID tuning).
\end{itemize}

\subsection{PID Tuning}
The controller has 12 PID controllers for 12 states. These states are the x/y/z positions, the yaw/pitch/roll orientations, the x/y/z velocities, and the yaw/pitch/roll rates. Each controller has 3 parameters to tune. Values are set as described in "Using the App" and testing the controls can be done as described in "Basic Drone Testing". Below is a recommended process for tuning the drone.
\begin{figure}[H]
    \centering
    \includegraphics[width=1\textwidth]{img/IMG_3141}
    \caption{Quadcopter Control Model}
    \label{fig:ControlModel}
\end{figure}
\begin{enumerate}
\item On-apparatus tuning:
\begin{enumerate}
\item Set all PID values to 0.
\item Set FLIGHT\_TYPE to 0 in test.py.
\item Test the controller to verify the drone and sensors work properly (all 4 motors should apply the same force). This can help catch other issues that may affect tuning.
\item Tune pitch controller:
\begin{enumerate}
\item Set pitch controller's P to 4 (this seems to be a good value)
\item Gradually increase and test the pitch rate controller's P until just below the point where oscillations occur. When running, hold the apparatus to prevent movement in other axes of rotation.
\item Gradually increase and test the pitch rate controller's I to correct steady-state error without causing slow oscillations.
\item Set the pitch rate controller's D as needed.
\item Proceed to tune the pitch controller's I and D.
\end{enumerate}
\item Tune roll and yaw controllers the same way pitch was tuned.
\end{enumerate}
\item Off-apparatus tuning.
\begin{enumerate}
\item Set FLIGHT\_TYPE to 1 in test.py.
\item Tune the elevation controller using a process similar to the pitch tuning process (be ready to type ctrl+c if something goes wrong).
\item Tune the x and y controllers similarly, doing so near a wall.
\end{enumerate}
\end{enumerate}
Some things to keep in mind when tuning:
\begin{itemize}
\item The PID struct has limits on I and on the output.
\item The sensor data will be noisy, which can make the D component too strong even with a filter.
\end{itemize}

\subsection{Frame Development}

Since it is free for students, Fusion360 was used for the frame design. Because of this, it is simplest to use Fusion for continued frame development. STL files have also been included in the \texttt{frame} repository for printing convenience or for editing in another modeling software.

\subsection{PCB Development}
Printed circuit board development can be accomplished easily using \href{https://easyeda.com/}{EasyEDA}. This software allows for design of a schematic and then layout of the circuit board. There is also a web interface that works quite well.

The current board revision files can be found \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/main/pcb/v1/EasyEDA}{here}. The zipped project folder contains all of the past board revisions, but the most up-to-date is  \texttt{PCB\_FCL\_Drone\_v0.0.13\_FAB}.
The current board revision is out of date in a few areas:
\begin{itemize}
\item The current power regulator (\href{https://github.com/Team-Wingfeather/fcl_senior_design/blob/main/resources/datasheets/C109322-TPS63070RNMR.pdf}{TPS63070}) is weak and burns out. This component will need to be replaced in the next version of the board, potentially by a \href{https://jlcpcb.com/partdetail/HoltekSemicon-HT75331/C14289}{HT7533-1} LDO. This is comparable to what is currently soldered onto the board. Note: there are multiple capacitors, resistors, and inductors that are used only by the previous component and will also need to be removed. See the \href{https://github.com/Team-Wingfeather/fcl_senior_design/blob/main/resources/datasheets/C109322-TPS63070RNMR.pdf}{datasheet} and \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/main/pcb/v1/EasyEDA}{EasyEDA} files for which components to replace.
\item The bottom ToF sensor leaves little room for a velcro strap, so it could be moved, or the frame CAD file could be modified to accommodate.
\item Resistor 13 on the board has been halved by a soldered-on parallel resistor of equal value. This creates a 1/3 voltage divider, bringing the 2C 7.4V battery voltage into a readable range for the ESP32. Thus, on board version 2, R13 should be replaced with a 50K Ohm resistor.
\end{itemize}
Some other optional changes include:
\begin{itemize}
\item The green LED is painfully bright and could use a larger resistor to dim it.
\item The ToF interrupt pins could be connected to the ESP32 for marginally faster sensor response.
\item The MPU6050 IMU could be moved to its own I2C line.
\item GPIO 23 could be switched to GPIO 16. The motor 4 PWM pin failed in testing and was swapped to pin 16. This should likely be made a permanent hardware change. See \href{https://github.com/Team-Wingfeather/fcl_senior_design/commit/29bd241f75058da32211addb992e50635beb5c23}{this commit} that patches the software.
\end{itemize}

\subsection{App Development}

The companion app is built as a Python application in the Qt framework. It is located in the \texttt{wingfeather\_app} repository, with its entry point in \texttt{main.py}. 

The main thing to keep in mind when developing the application is that since it is written in Python, dependencies and necessary files and directories must be bundled with the application, often creating complex file paths.

Additionally, the app is currently configured to bundle binaries from the \texttt{config} subdirectory and not the main repository. This allows for selection of the desired firmware, but is a disconnect between the app and the main repository that could be changed in the future. Because of this, when the firmware is updated, the binaries in this directory must also be changed.
