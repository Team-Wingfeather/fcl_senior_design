\subsection{Assembling the components}

Once the necessary components have been obtained, assembling the drone is fairly straightforward.
\begin{enumerate}
\item Solder the battery connector of choice to the bottom of the board. The plastic part of the connector can slide off of the metal part, so it can be helpful to super-glue the plastic to the PCB before soldering. Note that there are two connector options: JST-PH and JST-XH. To date, the team has been using JST-PH connectors for their minimal profile. Be careful to match the polarity of the batteries to the polarity of the board, marked with $+/-$ symbols near the connector.

\end{figure}
\item Using header pins, solder 2 time-of-flight sensor daughter boards onto the main board---one on top and the other on the bottom. These should use the 6-pin header slots, and be soldered onto the face with the printed labels.

\item Place the PCB on the frame and use 7 M2 screws to connect them. Note that the orientation of the PCB is determined by which corner does not have a hole for a screw. The front of the drone is the side with the USB connection.

\item Connect the propellers onto the motors. Make sure to select where each motor will go and match the propeller direction with the direction indicated around the screw nearest to that motor. Pushing gently onto the table can help affix the propellers to the shafts. Finally, it  is conventional to use a different color of propellers to designate the front of the drone, depending on the model purchased.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{img/IMG_4407}
    \caption{Control Loop for Quad-copter Flight}
    \label{fig:myimage}
\end{figure}
\item Slide the 4 motors into the designated recesses. It may prove easier to solder the motor connections before gluing the motors in.
\item Solder each motor to the nearest 2 through holes on the circuit board. Note that polarities of the annular rings alternate around the drone. This is so that the motors across from each other spin the same direction and adjacent motors spin opposite directions. In practical terms, the square annular ring should always be connected to the same motor terminal ``polarity'' (designated by the color of the wire). Which color may need to be determined by trial and error. For the current motors, the white motor lead should be soldered to the circular annular ring. In the end, make sure the direction each motor spins matches the arrow located around the nearest PCB screw. (This can be verified by flashing the drone using the app and connecting the battery---each motor should spin briefly). Note that it may be helpful to twist the wires from each motor before soldering. After soldering, this twisted pair may be hot glued to the frame to save space. Use caution when using the glue gun and soldering iron, as they can easily melt the 3D-printed frame.

\subsection{Programming the Drone}

\subsection{Using the App}
As mentioned in the Quick Start section, the app can be downloaded from the project \href{https://github.com/Team-Wingfeather/fcl_senior_design}{GitHub page} under \href{https://github.com/Team-Wingfeather/fcl_senior_design/releases}{releases}. Details on the features of each tab are listed below.
\subsubsection{Graphing Tab}


\subsubsection{Planning Tab}
The planning tab includes multiple stubs for lab options to be implemented in the future. Current progress on the project has been solely towards simple PID altitude control, but this tab will be more useful as the lab grows more complex.

\subsubsection{Programming Tab}
The programming tab is for setting the PID values to flash to the drone. Currently, the parameter-setting function of this tab is unimplemented. In the future, it should be able to set the altitude position P, I, and D values within a reasonable range that is determined experimentally. Additionally, the advanced mode (discussed in a later section) could be augmented to allow for full parameter access.

The ``Flash Drone'' button is functional, and will auto-detect the drone and flash scripts and parameters. These parameters are currently located in a \texttt{config} directory that is created the first time the drone is flashed. Once this is completed, the developer may enter this directory to modify the parameters for flashing. The \texttt{commands.txt} file is currently unused, but an in-progress implementation of the scripting exists on the branch \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/commander-control}{\texttt{commander-control}}.

Note that the drone hardware has been designed so that the user can plug the drone into the computer for flashing without disconnecting the battery. 

There are a few known minor bugs with the flashing system, but disconnecting and reconnecting the drone should solve most of these. In general, do not press start while the drone is connected to the computer.

Flashing the drone with modified firmware via the app can only be accomplished via advanced mode (see below). Each year the lab is performed should only need one version of the firmware, so this simplifies the setup. Changes for automation of the versioning system on this could be made in the future.



\subsubsection{Advanced Mode}
The app has an advanced mode that can be accessed via the file tab on the menu bar. The purpose of this mode is to allow users to use more features of the drone for debugging, development, and advanced use beyond lab experiments. Currently, this mode only adds an option to the programming tab to allow the user to force a full flash of the drone without regard to the current partition table.

\section{Drone Development}

\subsection{Development Environment}
The current framework for developing firmware for the Wing Feather is \href{https://docs.espressif.com/projects/esp-idf/en/v5.5.4/esp32/get-started/index.html}{ESP-IDF v5.5.4}. Espressif has recently released IDF v6 and the project could be upgraded to v6 if desired. An installation guide can be found by following that link.

\subsection{IDE Setup}
For ease of use, you can develop in VSCode with the \href{https://github.com/espressif/vscode-esp-idf-extension/blob/master/README.md}{ESP-IDF Extension}. 
The extension adds buttons to the VSCode toolbar that allow for easy compilation and flashing of the firmware.

If you have a preferred code editor though, the command line experience is very easy.

\subsection{Basic Repository Structure}

\subsubsection{The drone-firmware directory}
The entire esp-idf project for the drone firmware can be found in the drone-firmware directory. The directory contains the following files:

\begin{itemize}
  \item A CMakeLists.txt file that specifies the project's name and build options.
  \item A partition table that specifies the flash layout of the project.
  \item A main directory
  \begin{itemize}
    \item A CmakeLists.txt file that specifies the entry point, component directory, and required headers for main.
    \item A main.c file that contains the entry point
    \item A linker\_fragment.lf file that specifies some linker rules. This is so that some crazyflie functions get linked correctly in esp-idf.
    \item An idf\_component.yml file that specifies the dependencies of the project.
  \end{itemize}
  \item A components directory that contains all of the components used by the project
  \item The rest of the files at the firmware root are autogenerated by the build system.
\end{itemize}

\subsection{Basic Drone Testing}
To test the controller:
\begin{enumerate}
\item If using the test apparatus, fasten the drone to the apparatus.
\item Place the drone upright and flat on the table (while still on the apparatus is applicable) and plug in the battery. If the drone is not perfectly level at initialization or during resets, the IMU will not start up properly. It is recommended to already have the plug near the appropriate place before laying
\item Flash the PID parameters as instructed in "Using the App" and press start. NOTE: Keep the drone flat in case resetting at the end of the flash messes up the IMU.
\item Connect to the drone's Wifi. (The password is "Password").
\item Run test.py. Press ctrl+c in the terminal to stop the drone. If something goes wrong and ctrl+c doesn't stop the drone, press the reset button on the drone.
\item To plot logged data, run plotcsv.py
\end{enumerate}
Setting up test.py:
\begin{itemize}
\item LOG\_TYPE: Selects which values you want to log.
\begin{itemize}
\item Additional sets of logged values can be made following the formatting of other sets.
\item In log\_callback(), print() and log.write() must be specified. flight\_test must also be updated to add a new set.
\item Values available to log can be found in LogableValues.txt.
\item There is a limit to how much you can log (usually no more than 6 floats)
\end{itemize}
\item JUST\_LOG: Set to 1 if you just want to gather values without running the motors.
\item FLIGHT\_TYPE: Set to 1 for normal flight. Setting to 0 will specify a target thrust, yaw, pitch, and roll (useful for PID tuning).
\end{itemize}

\subsection{PID Tuning}
The controller has 12 PID controllers for 12 states. These states are the x/y/z positions, the yaw/pitch/roll orientations, the x/y/z velocities, and the yaw/pitch/roll rates. Each controller has 3 parameters to tune. Values are set as described in "Using the App" and testing the controls can be done as described in "Basic Drone Testing". Below is a recommended process for tuning the drone.
\begin{enumerate}
\item On-apparatus tuning:
\begin{enumerate}
\item Set all PID values to 0.
\item Set FLIGHT\_TYPE to 0 in test.py.
\item Test the controller to verify the drone and sensors work properly (all 4 motors should apply the same force). This can help catch other issues that may affect tuning.
\item Tune pitch controller:
\begin{enumerate}
\item Set pitch's P to 4 (this seems to be a good value)
\item Gradually increase and test pitch rate's P until just below the point where oscillations occur. When running, hold the apparatus to
\item Gradually increase and test pitch rate's I to correct steady-state error without causing slow oscillations.
\item Set pitch rate's D as needed.
\item Proceed to tune pitch's I and D.
\end{enumerate}
\item Tune roll and yaw controllers the same way pitch was tuned.
\end{enumerate}
\item Off-apparatus tuning.
\begin{enumerate}
\item Set FLIGHT\_TYPE to 1 in test.py.
\item Tune the elevation controller using a process similar to the pitch tuning process (be ready to type ctrl+c if something goes wrong).
\item Tune the x and y controllers similarly, doing so near a wall.
\end{enumerate}
\end{enumerate}
Some things to keep in mind when tuning:
\begin{itemize}
\item The PID struct has limits on I and on the output.
\item The sensor data will be noisy, which can make the D component too strong even with a filter.
\end{itemize}

\subsection{Frame Development}

\subsection{PCB Development}
Printed circuit board development can be accomplished easily using \href{https://easyeda.com/}{EasyEDA}. This software allows for design of a schematic and then layout of the circuit board. There is also a web interface that works quite well.

The current board revision files can be found \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/main/pcb/v1/EasyEDA}{here}. The zipped project folder contains all of the past board revisions, but the most up-to-date is  \texttt{PCB\_FCL\_Drone\_v0.0.13\_FAB}.
The current board revision is out of date in a few areas:
\begin{itemize}
\item The current power regulator (\href{https://github.com/Team-Wingfeather/fcl_senior_design/blob/main/resources/datasheets/C109322-TPS63070RNMR.pdf}{TPS63070}) is weak and burns out. This component will need to be replaced in the next version of the board, potentially by a \href{https://jlcpcb.com/partdetail/HoltekSemicon-HT75331/C14289}{HT7533-1} LDO. This is comparable to what is currently soldered onto the board. Note: there are multiple capacitors, resistors, and inductors that are used only by the previous component and will also need to be removed. See the \href{https://github.com/Team-Wingfeather/fcl_senior_design/blob/main/resources/datasheets/C109322-TPS63070RNMR.pdf}{datasheet} and \href{https://github.com/Team-Wingfeather/fcl_senior_design/tree/main/pcb/v1/EasyEDA}{EasyEDA} files for which components to replace.
\item The bottom ToF sensor leaves little room for a velcro strap, so it could be moved, or the frame CAD file could be modified to accommodate.
\item Resistor 13 on the board has been halved by a soldered-on parallel resistor of equal value. This creates a 1/3 voltage divider, bringing the 2C 7.4V battery voltage into a readable range for the ESP32. Thus, on board version 2, R13 should be replaced with a 50K Ohm resistor.
\end{itemize}
Some other optional changes include:
\begin{itemize}
\item The green LED is painfully bright and could use a larger resistor to dim it.
\item Connecting the ToF interrupt pins to the ESP32 for marginally faster sensor response.
\item Moving the MPU6050 IMU to its own I2C line.
\item Switching GPIO 23 to GPIO 16. The motor 4 PWM pin failed in testing and was swapped to pin 16. This should likely be made a permanent hardware change. See \href{https://github.com/Team-Wingfeather/fcl_senior_design/commit/29bd241f75058da32211addb992e50635beb5c23}{this commit} that patches the software.
\end{itemize}
